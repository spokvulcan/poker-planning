# A roster exit reconciles the auto-reveal countdown through the round module; admission and identity merges do not

**Status:** accepted

The **auto-reveal countdown** arms once every **non-spectator** has voted. That predicate (`Votes.areAllVotesIn`) reads two sets тАФ the room's votes *and* its non-spectator roster тАФ so a change to *either* must reconcile the countdown. Reconciliation (`Countdown.evaluate`) was wired only at the two vote-write sites in the **voting round** module (`castVote`, `retractVote`). But the roster also shrinks elsewhere: becoming a **spectator** (`editUser`) and leaving or being **removed** (`leaveRoom`) each delete votes and drop a non-spectator, and neither re-evaluated. Result: when the last non-voter became a spectator or left, the room was now fully voted but the countdown never armed тАФ auto-reveal silently never fired, and stayed stuck until some unrelated vote write happened to re-trigger evaluation.

The fix makes the round module the *sole writer of the votes table* for these paths тАФ the half of [ADR-0002](0002-voting-round-is-an-orchestration-module.md) the code had not yet built. A new round operation, **`dropVoter`**, deletes a departing or spectating member's votes and reconciles the countdown in one step. `editUser` and `leaveRoom` write the membership change (the spectator flip or the membership row тАФ identity is theirs to own) and then hand off to `dropVoter`. The kick path (`remove`) already funnels through `leaveRoom`, so it is covered.

## Considered Options

- **`dropVoter` as a sole-mutator operation** (chosen). The vote deletion moves out of the users module and behind one round operation, so a roster exit cannot change who-can-vote without reconciling the countdown. The invariant gains a single home, and the failure mode тАФ a new roster-mutating site forgetting to reconcile тАФ is structurally removed for the vote-deleting paths.
- **A thin `reconcile(roomId)` called at each site** (rejected). Add one function and call it from every roster-mutating handler. It fixes the present bug but keeps the same interface тАФ "remember to call it" тАФ so the next roster mutation re-opens the gap. By the deletion test it concentrates nothing; it is a renamed pass-through to `evaluate`.
- **Re-check the predicate at reveal time** (rejected). Have the scheduled reveal no-op if not all-in when it fires. This guards only the over-reveal direction and does nothing for the stuck-countdown bug (no armed job exists to re-check); it also contradicts the latecomer decision below.

## Consequences

- **Admission is deliberately not symmetric.** A member who appears *after* the countdown is armed тАФ joining (`joinRoom`) or ceasing to be a spectator (`editUser`) тАФ does **not** cancel it; the scheduled reveal still fires even though that member has not voted ("let it reveal"). Admission only grows the roster and needs no `dropVoter` тАФ but that is sound *only* because **a re-admitted member never already holds a vote** (next bullet). Were that not enforced, un-spectating a member who already had a vote row could silently complete the round without arming тАФ the same stuck-countdown class this ADR set out to kill. A future architecture review will see admission not reconciling and should read it against the votelessness invariant, not as a leak.
- **Spectators are voteless тАФ an enforced invariant.** No `votes` row ever belongs to a spectator, guaranteed at the two write sites that could otherwise create one: the round refuses a spectator's ballot (`castVote` throws), and becoming a spectator deletes any vote held (`dropVoter`). The identity merge upholds it too тАФ `linkAnonymousToPermanent` *drops* rather than transfers a vote whose destination is a spectator. Note the weaker claim "admission never touches the votes table" is **not** enough on its own: an admitted member could already hold a stray vote (a spectator's ballot, or a merge that re-points a vote onto a spectator). Enforcing votelessness at those sites is what makes the admission asymmetry above safe.
- **The identity merge does not reconcile the countdown.** `linkAnonymousToPermanent` re-points vote *ownership* during a sign-in account merge тАФ identity plumbing, not a round action, on a cold path тАФ and writes the votes table directly without calling `evaluate`. Cost: a merge that genuinely completes a live round (the anonymous duplicate was the last blocker) won't auto-arm until the next vote or roster change. It does uphold the votelessness invariant (dropping, not transferring, a vote onto a spectator destination). Recorded so the non-reconciliation isn't mistaken for a sole-writer violation (mirrors how [ADR-0003](0003-demo-is-a-client-simulation.md) carves the demo out of ADR-0002).
- **`dropVoter` reuses the symmetric `Countdown.evaluate` unchanged.** Removing a member from the roster can complete the round but cannot create a new non-voter, so the re-evaluation only ever arms тАФ except the degenerate case where the last voter is gone (roster empty тЖТ not all-in), which `evaluate` correctly cancels. No new countdown primitive.
- **Ordering is now a local obligation:** callers write the membership change first, then call `dropVoter`, so `areAllVotesIn` reads the new roster. This is far weaker than the previous cross-module "remember to call `evaluate` wherever you touch votes or memberships."
- **The bug path becomes testable through the round's interface.** "Last non-voter becomes a spectator тЖТ countdown arms" and "тАжleaves тЖТ arms" are now one assertion each against the round module тАФ paths that had no round-module entry before, which is why the bug hid.
